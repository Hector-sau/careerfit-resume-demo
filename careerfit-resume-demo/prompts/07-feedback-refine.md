# 07 反馈调整 Prompt

## 使用场景

当用户已经拿到 AI（豆包、千问、DeepSeek、ChatGPT、Claude、Kimi 等任意工具）生成的简历优化结果，但对某部分不满意时，用这个 Prompt 生成一份带着具体修正要求的"调整版"Prompt，让用户带着原始 JD、原始背景和上一次的不满意输出，回到同一个 AI 对话里继续修正，而不用从头重新描述一遍需求。

网页表单版（`index.html`）里的"第 4 步：对结果不满意？"就是这个 Prompt 的可视化实现：用户粘贴不满意的输出、勾选问题类型（对应下面的分类），网页在本地拼出这份调整 Prompt。

## 设计原则

- 不猜测用户哪里不满意，问题类型必须由用户自己勾选或填写，工具只负责把选择翻译成具体、可执行的修正指令。
- 问题分类直接对齐 `docs/evaluation-method.md` 里定义的错误类型（漏提取、误提取、分类错误、匹配虚高、建议泛化、经历幻觉），保证"发现问题的语言"和"评测记录的语言"是一套体系，方便后续把真实反馈整理进 `docs/failure-cases.md`。
- 调整 Prompt 里重新完整带上 JD 和个人背景，即使用户换了一个新对话（而不是接着原对话问），AI 也有足够上下文，不需要用户再手动复制一遍。
- 只要求 AI"修正被指出的问题"，不要求整体重写，减少不必要的内容变动。

## 问题分类与对应修正指令

| 用户勾选的问题（网页文案） | 对应错误类型 | 生成的修正指令 |
| --- | --- | --- |
| 内容太笼统，缺具体细节 | 建议泛化 | 要求增加具体细节（场景、方法、数量、对象），不要停留在抽象描述 |
| 好像编造了我没提供的经历/数据 | 经历幻觉 | 要求删除或标注所有未在个人经历里提供过的具体事实、机构名、数据，标注为"建议补充" |
| 建议和简历内容没分开 | 分类错误 | 要求严格按 A/B 两部分重新整理，并标注 A 部分哪些是保守版、哪些是增强版 |
| 岗位要求提取漏了重点 | 漏提取 | 要求重新阅读完整 JD 原文，补充遗漏的核心职责、必备技能或加分项 |
| 把不重要的内容当成了核心要求 | 误提取 | 要求去掉 JD 原文没有明确提到的内容，不要把加分项当成必备项 |
| 匹配度打分缺乏依据 | 匹配虚高（或虚低） | 要求评分更保守，且每项都要写清楚具体依据 |
| 求职建议太空泛，不好执行 | 建议泛化 | 要求补齐项目建议和下一步行动清单写清楚具体要做什么、交付物是什么 |
| 没有说明信息来源 | （信息来源缺失，属于 06 Prompt 的第 8 条原则未被遵守） | 要求补充信息来源与判断依据表格 |
| 格式/结构看不懂 | （格式问题，不在原六类错误类型内，属于网页版新增的可用性维度） | 要求严格按 A、B 两部分和编号小标题重新输出，多用表格 |

## Prompt（网页表单版会自动拼装，这里是对应的手动版模板）

```text
我之前根据下面这份 JD 和个人背景，让 AI 帮我做简历优化和求职建议，但对上一次的输出不满意，需要你在同一份背景下重新调整。

【目标岗位 JD】
<<<
粘贴你之前用的 JD
>>>

【我的个人背景】
- 学历/专业：
- 目标岗位：
- 过往实习：
- 项目经历：
- 校园/社团经历：
- 内容/运营经历：
- 会使用的工具：
- AI 工具使用经历：
- 当前简历里想优化的内容：

【AI 上一次生成的内容，我不满意】
<<<
粘贴 AI 上一次的输出（可以只贴有问题的那部分）
>>>

【这次需要你重点修正的问题】
- 从上面的问题分类表里选择适用的几条，或者自己描述

【调整要求】
1. 请针对上面指出的问题重新生成，不要改动没有被指出问题的部分。
2. 修正后仍然要遵守最初的规则：不允许编造我没有提供过的经历；缺证据的地方写“建议补充”，不要虚构数字；不承诺我一定能拿 offer；简历内容和求职建议要分开，不要把建议写成我已经完成的经历；补齐项目只提供候选方向，由我自己决定；每条关键求职建议都要标注信息来源。
3. 请仍然按照 A. 简历优化结果 / B. 求职建议 的结构重新输出被修正的部分，能用表格的地方继续用表格。
4. 如果我指出的问题里有互相矛盾的地方，请指出矛盾并让我确认，不要自己猜测处理。
```

## 评测重点

- 修正后的输出是否真的解决了用户勾选的问题，而不是泛泛重写一遍。
- 是否意外改动了用户没有提出异议的部分。
- 是否仍然遵守不编造经历、简历内容和求职建议分离等基础原则。
- 用户反复调整同一个问题超过 2-3 轮仍未解决时，应该提示用户这可能是 JD 或背景信息本身不够具体，而不是无限重试。

## 和评测体系的关系

用户在网页上勾选的问题类型，理论上就是最真实的失败案例来源。后续如果收集到有代表性的真实调整案例（哪怕只是自己测试时发现的），建议按 `docs/failure-cases.md` 的模板记录下来，形成"用户反馈 -> 失败案例 -> Prompt 迭代"的闭环，这也是这个功能除了提升单次使用体验之外，更长期的产品价值。
